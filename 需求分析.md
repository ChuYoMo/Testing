## 1. 引言

### 1.1 编写目的

本文档面向《软件测试》课程实验项目"简单去中心化加密货币交易所（DEX）核心系统与可视化 GUI"，从用户、业务和系统三个角度梳理项目应当具备的功能与非功能需求，为后续架构设计、详细设计、编码实现及测试用例编写提供基线依据。

### 1.2 项目背景

去中心化加密货币交易所（DEX）是区块链生态中的典型业务系统，涉及账户认证、资产托管、订单簿撮合、链式账本等多种知识点，适合作为教学演示和软件测试实验对象。本项目以 Python 标准库实现一个**纯演示用、单进程、单文件持久化**的 DEX 原型，不连接真实区块链网络、不管理真实私钥、不处理真实资金。

> **关于 "DEX" 名称的说明**：本项目仅模拟 DEX 的"订单撮合 + 链式账本"思想，系统架构上仍是**教学用中心化单机原型**——撮合由单一进程完成、钱包资产由系统集中托管、链式账本不参与共识也不进行 P2P 传播。命名"DEX"主要服务于教学语境下的概念对照，而非声称项目具备真实的去中心化属性。

### 1.3 术语定义

| 术语 | 含义 |
| --- | --- |
| DEX | Decentralized Exchange，去中心化交易所。本项目仅为教学模拟，不具备真实去中心化属性 |
| 限价单（LIMIT） | 用户指定价格与数量的订单，仅在价格满足条件时才能成交 |
| 市价单（MARKET） | 用户只指定数量、不指定价格的订单，按对手簿当前最优价格立即成交 |
| 订单簿（OrderBook） | 按价格优先、时间优先排序的买卖订单容器 |
| 撮合（Matching） | 将买卖订单按规则配对成交并完成资金结算的过程 |
| 自成交保护 | 防止同一用户的买卖订单互相成交 |
| 冻结余额 | 已被挂单占用、不可再次使用的资产 |
| 链式账本 | 通过区块哈希链接来维护数据不可篡改的简化区块链 |
| Bearer Token | HTTP `Authorization` 头中的会话令牌格式：`Bearer <token>` |

## 2. 总体目标

### 2.1 项目定位

本项目是**教学演示原型**，定位如下：

1. 在不依赖任何第三方库的前提下，提供一个能跑通"注册—充值—下单—撮合—成交—上链"完整业务闭环的 DEX 系统。
2. 代码结构清晰、模块解耦，便于编写单元测试与集成测试。
3. 提供命令行演示和浏览器 GUI 两种交互入口，方便课程演示。
4. 数据全部落盘到单一 SQLite 文件，保证演示结果可复现、可重置。

### 2.2 核心目标

| 编号 | 目标 | 衡量方式 |
| --- | --- | --- |
| G1 | 提供完整的用户认证流程（注册、登录、登出、token 校验） | 自动化测试 `AuthServiceTest`、`HttpAuthIntegrationTest` 全部通过 |
| G2 | 提供多币种、多交易对的限价/市价撮合能力 | 自动化测试 `test_required_btc_and_eth_trading_flow`、`test_market_*` 全部通过 |
| G3 | 资产冻结与成交结算金额精度无误差 | `Decimal` 8 位小数量化、`test_decimal_normalization_rejects_non_finite_values` 通过 |
| G4 | 提供链式账本记录所有成交，支持完整性校验 | `validate_chain()` 在演示流程中持续返回 True |
| G5 | 提供浏览器 GUI 与 JSON API，所有 GUI 可执行操作均能通过 API 完成 | `gui.py` 启动后能在全部 11 个 API 接口（`/api/state`、`/api/register`、`/api/login`、`/api/logout`、`/api/wallet`、`/api/order`、`/api/cancel`、`/api/seal`、`/api/validate`、`/api/demo-flow`、`/api/reset`）上正常完成操作 |
| G6 | 系统状态可重置、可复现演示数据 | `/api/reset` 能清空所有表并重新种入示例账户 |

### 2.3 非目标

下列内容**不在**本项目目标范围内，不予实现：

- 不接入真实区块链、不实现共识算法、不进行 P2P 通信。
- 不管理真实私钥、不进行链上签名验证。
- 不实现手续费模型、滑点模型、撮合费率分级。
- 不支持高并发场景下的多机/多进程协调（仅依赖 SQLite 内置写锁）。
- 不实现 KYC、风控、审计、洗钱合规等监管功能。
- 不实现 token 过期、刷新、多端互踢策略。

## 3. 用户与场景

### 3.1 用户角色

| 角色 | 说明 |
| --- | --- |
| 演示用户（Trader） | 通过 GUI 或 CLI 演示交易流程的角色，对应代码中 `alice`、`bob`、`carol` 等账户 |
| 教师/评审 | 通过浏览器 GUI 直观查看撮合、成交、链结构，验证教学目标 |
| 测试人员 | 通过 `unittest` 调用底层服务编写黑盒/白盒测试 |
| 开发者 | 阅读源码理解 DEX 业务模型，扩展或修改业务规则 |

### 3.2 典型业务场景

#### 场景 1：教学演示（一键流程）

教师在课堂上启动 GUI，点击"演示流程"按钮一键执行 Bob/Alice/Carol 交易场景，观察订单簿、成交记录、钱包余额、区块链结构的变化，并完成链完整性校验。

#### 场景 2：手工交易演示

学生在 GUI 中切换不同用户登录，分别提交限价单和市价单，观察资金冻结、解冻、成交结算与撤单后的余额恢复。

#### 场景 3：自动化测试

测试人员运行 `python3 -m unittest discover -v`，验证核心交易流程、市价单跨档成交、自成交保护、流动性不足拒绝、Token 鉴权等业务规则的正确性。

#### 场景 4：API 集成

外部脚本通过 `curl`、`requests` 等工具调用 JSON API，演示 DEX 系统作为后端服务被前端或其他系统集成的能力。

### 4.2 详细功能描述

#### 4.2.1 用户认证（F-AUTH）

- **F-AUTH-01 注册**：用户名经 `strip()` 后不为空且未注册，密码不为空。重复用户名抛 `UserAlreadyExistsError`。
- **F-AUTH-02 登录**：核对密码哈希（`secrets.compare_digest`），生成 32 位 hex token 并写入 `sessions` 表。
- **F-AUTH-03 登出**：从 `sessions` 表删除 token，对应内存索引同步移除。
- **F-AUTH-04 鉴权**：钱包、下单、撤单、登出均要求 `Authorization: Bearer <token>`，操作主体只取 token 解析的用户，请求体中夹带的 `username` 一律忽略。
- **F-AUTH-05 密码安全**：默认 PBKDF2 迭代 120 000 次，盐值 16 字节随机。

#### 4.2.2 钱包账户（F-WAL）

- **F-WAL-01 多币种**：仅支持 `Asset.BTC`、`Asset.ETH`、`Asset.USDT`，未注册币种抛 `UnsupportedAssetError`。
- **F-WAL-02 充值/提现**：金额必须为正且为合法 `Decimal`（拒绝 NaN/Infinity）。提现时可用余额不足抛 `InsufficientBalanceError`。
- **F-WAL-03 冻结相关**：冻结从可用扣减、增加冻结；解冻反之；消耗冻结只扣减冻结，不增加可用，用于成交结算。冻结/解冻余额不足抛 `InsufficientFrozenBalanceError`。
- **F-WAL-04 快照**：返回 `{owner, available, frozen, total}` 字典，金额以字符串形式表达 `Decimal`。

#### 4.2.3 订单撮合（F-ORD）

- **F-ORD-01 限价单**：买单冻结 `价格 × 数量` 计价资产，卖单冻结基础资产数量。买价 ≥ 对手卖价 / 卖价 ≤ 对手买价时成交，成交价取 maker 价格。买方限价高于成交价的差额自动解冻返还。
- **F-ORD-02 市价单**：先预走对手簿（剔除自己挂单），可成交数量小于请求数量直接抛 `EmptyOrderBookError` 且不冻结资金；流动性充足时按精确所需总额冻结资金，剩余直接置 `CANCELLED`。
- **F-ORD-03 排序规则**：买盘按价格降序、同价按 `sequence` 升序；卖盘按价格升序、同价按 `sequence` 升序。
- **F-ORD-04 状态机**：`OPEN → PARTIALLY_FILLED → FILLED`，或 `OPEN/PARTIALLY_FILLED → CANCELLED`。状态由 `Order.refresh_status()` 根据剩余数量自动刷新。
- **F-ORD-05 自成交保护**：
    - 限价单：遇到同用户对手单立即停止本轮撮合，`self_trade_blocked=True`。
    - 市价单：跳过该 maker 继续撮合后续档位。
- **F-ORD-06 撤单**：仅订单所有者可撤单；非活跃订单（FILLED / CANCELLED）抛 `OrderNotCancellableError`；按方向解冻剩余资金，订单状态置 `CANCELLED`。
- **F-ORD-07 订单簿快照**：返回 `{buy: [...], sell: [...]}`，每个订单字典包含价格、数量、剩余数量、状态、序号、订单类型。

#### 4.2.4 链式账本（F-CHN）

- **F-CHN-01 创世区块**：链为空时自动创建 `index=0`、`previous_hash="0"*64`、含 `GENESIS` 描述的区块。
- **F-CHN-02 自动封块**：每次 `add_trade` 都将成交记录加入待打包区；待打包数 ≥ `block_capacity`（默认 2）时自动封块。
- **F-CHN-03 校验**：`validate_chain()` 重新计算每个区块哈希、检查与前一区块的 `previous_hash` 一致性，发现篡改抛 `ChainValidationError`。
- **F-CHN-04 快照**：`chain` / `pending_transactions` / `export_chain` 均深拷贝返回，外部代码无法直接修改内部状态。

#### 4.2.5 持久化（F-PER）

- **F-PER-01 SQLite**：`dex.db` 默认存储 7 张表：`users`、`sessions`、`wallets`、`orders`、`trades`、`blocks`、`pending_transactions`，启用 WAL + 外键。
- **F-PER-02 schema 迁移**：`_migrate_schema()` 通过 `PRAGMA table_info` 检测列缺失并自动 `ALTER TABLE`，例如新增 `order_type` 列。
- **F-PER-03 重置**：`clear_db()` 仅删除数据，不删除 schema。

#### 4.2.6 浏览器 GUI 与 JSON API（F-API）

| `/` | GET | 否 | 单页 HTML |

| `/api/register` | POST | 否 | 注册 |
| `/api/login` | POST | 否 | 登录返回 token |
| `/api/logout` | POST | 是 | 登出 |
| `/api/wallet` | POST | 是 | 充值/提现 |

## 5. 数据需求

### 5.1 主要数据实体

| 实体 | 关键字段 | 说明 |
| --- | --- | --- |
| User | username（PK）, password_hash, password_salt, created_at | 用户 |
| Session | token（PK）, username | 会话令牌 |
| Wallet | (owner, asset)（联合 PK）, available, frozen | 钱包余额 |
| Order | order_id（PK）, user_id, pair, side, price, quantity, remaining_quantity, status, created_at, sequence, order_type | 订单 |
| Trade | trade_id（PK）, buy_order_id, sell_order_id, buyer_id, seller_id, pair, price, quantity, quote_amount, timestamp | 成交记录 |
| Block | block_index（PK）, timestamp, previous_hash, block_hash, transactions(JSON) | 区块 |
| PendingTx | id（自增 PK）, tx_data(JSON) | 待打包交易 |

### 5.2 金额表示

- 所有价格、数量、余额内部使用 `Decimal`，存库使用字符串形式。
- 统一量化到 8 位小数，向下舍入（`ROUND_DOWN`），避免向上"凭空多算"金额。
- 输入若为 NaN / Infinity / 非数字字符串，立即抛 `ValidationError`。

### 5.3 状态枚举

- `OrderSide`：BUY、SELL
- `OrderStatus`：OPEN、PARTIALLY_FILLED、FILLED、CANCELLED
- `OrderType`：LIMIT、MARKET
- `Asset`：BTC、ETH、USDT

## 6. 非功能需求

### 6.1 可用性

- 浏览器 GUI 启动后默认监听 `http://127.0.0.1:8000`；端口被占用时自动尝试 8001~8019。
- GUI 默认开启自动重载，源码改动后会自动重启服务。
- 错误一律以 JSON `{ok:false, error:"..."}` 返回，前端显示中文错误。

### 6.2 性能

- 单进程演示原型，目标 QPS 不做硬性约束。
- 撮合主循环为简单线性扫描，订单簿规模在课堂演示量级（百级订单）下响应在亚秒级。
- SQLite 启用 WAL 减少读写互锁，但**不**承诺多进程并发安全。

### 6.3 可靠性

- 状态一致性分两个阶段表述，避免歧义：
    - **参数校验阶段**（金额、币种、交易对、用户、订单所有权等）失败时立即抛异常，**不修改任何钱包、订单、链状态**。
    - **撮合执行阶段**（资金冻结、`consume_frozen`、订单状态刷新、上链）当前**未做事务回滚**，若中途出现异常，可能出现部分状态写入（如已冻结但未结算）。教学场景下接受此限制，后续可通过显式 SQLite 事务包裹 `_match_order` 改进。
- 链式账本通过哈希校验提供基本的篡改检测能力。
- 订单簿、链、待打包交易均深拷贝返回，外部不能通过持有引用修改内部状态。

### 6.4 安全性

- 密码使用 PBKDF2-HMAC-SHA256（120 000 次迭代）+ 随机盐存储，不可逆。
- token 由 `secrets.token_hex(16)` 生成，使用恒等比较。
- 受保护接口操作主体只信任 token 解析结果，避免请求体伪造身份。
- 静态文件路径做 `commonpath` 校验，防止路径穿越。

### 6.5 可测试性

- 服务对象支持注入 `clock`、`password_hasher`、`conn`，便于在测试中构造确定性输入。
- 撮合引擎 / 区块链 / 钱包 / 认证服务均可在内存模式下独立工作（不依赖 SQLite）。
- 订单 ID（`O000001`）、成交 ID（`T000001`）由内置计数器生成，便于断言。

### 6.6 可维护性

- 单一职责：每个模块只承担一类业务，模块间通过显式构造函数注入依赖。
- 异常体系层次化：`DEXError → AuthError/WalletError/OrderError/BlockchainError`，便于上层统一翻译。
- 文档化：README + 软件设计说明书 + 测试用例表 + 本需求文档。

### 6.7 可移植性

- 仅依赖 Python 3.8+ 标准库。
- 适用 Windows、Linux、macOS。
- SQLite 文件可跨机器迁移。

## 8. 实现边界与限制

- 状态保存在 SQLite 文件中，删除文件即清空所有数据。
- 不实现真实区块链共识、P2P 通信、链上签名。
- 不管理真实钱包地址和私钥。
- **充值 / 提现仅为演示用的钱包调账能力**：直接修改可用余额，**不接入**真实链上充值确认流程，**不进行**提现签名与广播。GUI 中允许任意用户对自己充值，是教学便利的设定，与真实交易所的法币/链上入金通道无关。
- 没有手续费模型。
- 撮合执行阶段未做事务回滚（与 §6.3 表述一致）：参数校验阶段失败不会修改状态；执行阶段若中途异常可能产生部分状态写入。
- 鉴权仅支持 Bearer Token，**无** token 过期、刷新、多端互踢。
- 会话同时存在于内存与 SQLite，不做跨进程同步。

## 9. 后续扩展建议

按优先级排序：

1. 认证模块更全面的测试：重复注册、token 过期、多端互踢、并发登录。
2. 钱包模块独立单元测试：余额不足、冻结不足、非法金额、不支持币种。
3. 撮合模块独立单元测试：价格优先、时间优先、自成交保护、不支持交易对、撤单边界。
4. 区块链模块独立单元测试：空封块、自动封块、篡改交易、篡改哈希。
5. 功能扩展：手续费、历史订单查询、止损/止盈单、深度图、撮合事务化、多进程协调。
