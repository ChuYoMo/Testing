## 1. 引言

### 1.1 编写目的

本文档面向开发者与评审组，从系统总体结构、模块划分、数据流、部署形态等角度刻画 DEX 项目的架构方案。架构设计以《需求分析说明书》提出的功能与非功能需求为输入，为后续详细设计与编码提供蓝图。

### 1.2 设计原则

| 原则 | 说明 |
| --- | --- |
| **单一职责** | 每个模块只承担一种业务（认证、钱包、撮合、链账本、持久化、网络）。 |
| **依赖注入** | 服务在构造时显式接收依赖，便于在测试与生产之间切换实现。 |
| **业务-持久化分层** | 业务模型保留在内存对象中，SQLite 作为可选的持久化适配器；核心逻辑在无 `conn` 时仍能运行。 |
| **不可变快照** | 链、订单簿、待打包交易等共享状态通过 `deepcopy` 返回，避免外部代码篡改。 |
| **统一异常体系** | 业务异常派生自 `DEXError`，HTTP 层负责翻译为状态码。 |
| **零外部依赖** | 仅使用 Python 标准库，便于在课堂演示与评审环境部署。 |

### 1.3 关键约束

- 单进程运行（`http.server.ThreadingHTTPServer`），不引入消息队列、缓存中间件、外部 RPC。
- 数据存储仅 SQLite 单文件；必须支持纯内存模式以便测试。
- 鉴权统一基于 Bearer Token；操作主体只取 token 解析结果。
- 演示原型，不要求严格事务一致性，但须保证金额标准化、链完整性、状态机正确。

## 3. 模块设计

### 3.1 认证模块（auth.py）

**职责**：用户注册、登录、登出、token 解析、密码哈希。

**关键类**：

- `PasswordHasher`：PBKDF2-HMAC-SHA256，可配置迭代次数，默认 120 000 次。
- `AuthService`：
    - 内存索引：`_users: Dict[str, User]`、`_sessions: Dict[str, str]`
    - 可选 `conn` 参数，存在时启动从 `users` / `sessions` 表加载。
    - 注册写入 `users` 表；登录写入 `sessions`；登出删除。

**对外接口**：`register(username, password)`、`login(username, password) -> token`、`logout(token)`、`get_user_by_token(token) -> User`、`is_authenticated(token)`、`user_exists(username)`。

**抛出异常**：`UserAlreadyExistsError`、`UserNotFoundError`、`InvalidCredentialsError`、`ValidationError`。

### 3.2 钱包模块（wallet.py）

**职责**：账户余额管理与冻结结算。

**关键类**：

- `WalletService`：
    - 接受 `supported_assets: Iterable[Asset]`，限制可操作币种。
    - 内存索引：`_wallets: Dict[str, Wallet]`，每个 Wallet 含 `balances` 与 `frozen_balances`。
    - 可选 `conn` 参数，存在时按 `(owner, asset)` 单行写回。

**对外接口**：`create_wallet_for_user`、`get_wallet`、`get_wallet_snapshot`、`deposit`、`withdraw`、`freeze`、`unfreeze`、`consume_frozen`。

**抛出异常**：`UnsupportedAssetError`、`InsufficientBalanceError`、`InsufficientFrozenBalanceError`、`ValidationError`。

### 3.3 撮合模块（engine.py）

**职责**：限价/市价撮合、订单簿维护、撤单、自成交保护、流动性不足拒绝。

**关键类**：

- `OrderBook`：按 `(pair.symbol, OrderSide)` 分别维护买卖列表，添加时按价格优先 + 时间优先排序。
- `MatchingEngine`：
    - 注入 `AuthService`、`WalletService`、`SimpleBlockchain`、`supported_pairs`、可选 `conn`。
    - 自带计数器：`_order_sequence`、`_order_id_sequence`、`_trade_id_sequence`，重启时从 SQLite 最大值恢复。

**对外接口**：

- `place_limit_order(user_id, pair, side, price, quantity)`
- `place_market_order(user_id, pair, side, quantity)`
- `cancel_order(user_id, order_id)`
- `get_order(order_id)` / `list_orders()` / `get_order_book_snapshot(pair)`

**核心算法**：

1. **限价单**：先冻结资金 → 进入撮合主循环 → 成交结算 → 剩余进入订单簿。
2. **市价单**：先预走对手簿（剔除自己）→ 不足直接抛异常 → 精确冻结 → 撮合 → 剩余置 CANCELLED（理论上不应出现，作为兜底）。
3. **撮合主循环**：从对手簿头部按 `pointer` 顺序扫描，价格不交叉退出；遇到自单时按订单类型决定停止 / 跳过；满量则 pop。

**抛出异常**：`ValidationError`、`UnsupportedTradingPairError`、`EmptyOrderBookError`、`OrderNotFoundError`、`OrderNotCancellableError`。

### 3.4 链账本模块（blockchain.py）

**职责**：维护一条简化区块链，记录所有成交。

**关键类**：

- `SimpleBlockchain`：
    - `_chain: List[Block]`、`_pending_transactions: List[dict]`
    - 启动时从 `blocks` / `pending_transactions` 表恢复；若无创世区块则自动创建。
    - 容量到达 `block_capacity` 时自动封块。

**对外接口**：`add_transaction`、`add_trade`、`seal_pending_transactions`、`validate_chain`、`chain`、`pending_transactions`、`export_chain`。

**抛出异常**：`InvalidBlockError`、`EmptyBlockError`、`ChainValidationError`。

### 3.5 持久化模块（database.py）

**职责**：SQLite schema 创建、迁移与重置。

**关键函数**：

- `init_db(db_path)`：打开连接、设置 `row_factory=Row`、`PRAGMA journal_mode=WAL`、`PRAGMA foreign_keys=ON`、调用 `_create_tables` 与 `_migrate_schema`。
- `clear_db(conn)`：按依赖顺序删除所有表数据，保留 schema。
- `_migrate_schema(conn)`：通过 `PRAGMA table_info(orders)` 检测 `order_type` 列；不存在则 `ALTER TABLE` 添加。

**Schema 总览**：见需求分析 §5.1。

### 3.6 领域模型（models.py）

- 枚举：`Asset`、`OrderSide`、`OrderStatus`、`OrderType`。
- 实体：`User`、`Wallet`、`Order`、`Trade`、`Block`、`OrderPlacementResult`、`TradingPair`。
- 工具：`normalize_decimal`、`ensure_positive`、`DECIMAL_SCALE`、`ZERO`。
- `Order.refresh_status()` 集中维护订单状态机。
- `Block.compute_hash()` 集中维护区块哈希算法（`json.dumps(..., sort_keys=True)` + SHA-256）。

### 3.7 异常体系（exceptions.py）

```text
DEXError
├── ValidationError
├── AuthError
│   ├── UserAlreadyExistsError
│   ├── UserNotFoundError
│   └── InvalidCredentialsError
├── WalletError
│   ├── UnsupportedAssetError
│   ├── InsufficientBalanceError
│   └── InsufficientFrozenBalanceError
├── TradingPairError
│   └── UnsupportedTradingPairError
├── OrderError
│   ├── InvalidOrderError
│   ├── SelfTradePreventedError
│   ├── OrderNotFoundError
│   ├── OrderNotCancellableError
│   └── EmptyOrderBookError
└── BlockchainError
    ├── EmptyBlockError
    ├── InvalidBlockError
    └── ChainValidationError
```

GUI 层捕获 `DEXError` 转为 HTTP 400；`KeyError`、`ValueError` 转为 400；鉴权失败为 401；未知路径为 404。

### 3.8 接入层（gui.py）

**关键类**：

- `DEXWebState`：拥有数据库连接和所有服务，对外暴露业务方法（`register`、`login`、`wallet_action`、`place_order`、`cancel_order`、`run_demo_flow`、`seal_block`、`validate_chain`、`reset`、`snapshot`）。
- `DEXRequestHandler`（继承 `BaseHTTPRequestHandler`）：
    - `do_GET`：路由 `/`、`/static/*`、`/api/state`。
    - `do_POST`：路由 11 个 API；其中钱包、下单、撤单、登出经 `_authenticate_or_fail` 强制鉴权。
    - 鉴权方法：从 `Authorization: Bearer <token>` 解析，调 `STATE.auth_service.get_user_by_token`，失败返 401。
    - 静态文件：`_send_static` 进行 `commonpath` 校验防穿越。

**辅助机制**：

- `_build_server`：从 `8000` 起依次尝试至 `8019`，遇到 `EADDRINUSE` 则换端口。
- `_watch_sources` + `os.execv`：监听项目所有 `.py` 文件 mtime，发生变化时通过 `os.execv` 在原进程内重启。

## 4. 关键数据流

### 4.1 提交限价买单（成功成交）

```text
Browser ──POST /api/order, Bearer token──► DEXRequestHandler.do_POST
        ←────────────────────────── ok=true, data=snapshot

DEXRequestHandler
  └─► _authenticate_or_fail() ── token → user
  └─► STATE.place_order(payload, user)
        └─► MatchingEngine.place_limit_order(user, pair, BUY, price, qty)
              ├─► _validate_user / _validate_pair / normalize_decimal
              ├─► _freeze_order_funds → WalletService.freeze(user, USDT, price*qty)
              ├─► _match_order
              │     └─► 对每个 maker：
              │           ├─► _execute_trade(taker, maker)
              │           │     ├─► WalletService.consume_frozen(seller, BTC, qty)
              │           │     ├─► WalletService.deposit(buyer, BTC, qty)
              │           │     ├─► WalletService.consume_frozen(buyer, USDT, quote)
              │           │     ├─► WalletService.deposit(seller, USDT, quote)
              │           │     ├─► 返还差额（若 buy_price > exec_price）
              │           │     └─► 刷新双方订单状态
              │           └─► SimpleBlockchain.add_trade(trade) → 自动封块（达容量）
              ├─► 剩余仍活跃 → OrderBook.add(order)
              └─► 持久化：_persist_order(taker)、_persist_order(maker)、_persist_trade
        └─► STATE._record_order_flow(result)
```

### 4.2 提交市价买单（流动性不足）

```text
MatchingEngine.place_market_order
  ├─► _validate_user / _validate_pair / normalize_decimal
  ├─► 计算对手簿可成交量（剔除自己）
  ├─► available_qty < quantity → raise EmptyOrderBookError
  └─► gui.py 捕获 DEXError → HTTP 400 {ok:false, error:"对手方可成交数量..."}
      （此时未冻结任何资金）
```

### 4.3 鉴权失败

```text
Browser ──POST /api/wallet, 缺/错 Bearer token──► DEXRequestHandler
        ←──────── HTTP 401 {ok:false, error:"缺少 Authorization 头" / "会话令牌无效或已过期"}
```

### 4.4 链完整性校验

```text
gui.py /api/validate
  └─► SimpleBlockchain.validate_chain()
        └─► 对每个 Block：
              ├─► 重算 compute_hash() 与存储 block_hash 比较
              ├─► index==0 校验 previous_hash="0"*64
              └─► index>0 校验 previous_hash == 前块 block_hash
        └─► 任何失败 → ChainValidationError（HTTP 400）
```

## 5. 部署架构

### 5.1 形态

```text
┌──────────────────────────────────────────────────┐
│ 单台开发机 / 单台课堂演示机                      │
│  ┌────────────────────────────────────────────┐  │
│  │ Python 3.8+ 进程：python3 gui.py           │  │
│  │  ├─ ThreadingHTTPServer（HTTP/1.1）        │  │
│  │  ├─ DEXWebState                            │  │
│  │  └─ Watcher 线程（自动重启）               │  │
│  └────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────┐  │
│  │ 静态资源 static/index.html app.js style.css│  │
│  └────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────┐  │
│  │ SQLite 文件 dex.db（WAL + shm）            │  │
│  └────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

- 单进程多线程：`ThreadingHTTPServer` 为每个连接派生线程，SQLite 通过 `check_same_thread=False` 共享连接。
- 数据库文件就地存储：默认 `dex.db`，删除即清空。
- 浏览器端为纯静态资源，状态保存在 `localStorage`（仅 token 与当前用户）。

### 5.2 启动流程

1. `gui.py` `main()` 解析 `--no-reload` 开关。
2. `DEXWebState.__init__` 初始化数据库与服务，无用户时种入示例账户。
3. `_build_server()` 选择可用端口。
4. 若开启重载，启动 watcher 线程轮询 `.py` 文件 mtime。
5. `serve_forever()` 阻塞主线程。
6. 收到变更或 `Ctrl+C` 时关闭服务；变更场景下 `os.execv` 重启进程。

### 5.3 进程内并发

- 业务服务对象内的状态修改（订单簿、钱包、链）依赖 GIL + SQLite 写锁；不显式加锁。
- 演示场景下不期望多浏览器同时高并发提交订单；如果出现，可能产生轻微竞态（已在限制条款中声明）。

## 7. 安全设计

| 风险 | 应对策略 |
| --- | --- |
| 密码泄露 | PBKDF2-HMAC-SHA256 + 16 字节随机盐；只存 hex 哈希，不存明文 |
| 会话伪造 | `secrets.token_hex(16)` 生成令牌；恒等比较（`secrets.compare_digest`） |
| 越权操作 | 服务端只信任 token 解析的用户，请求体 `username` 一律忽略 |
| 路径穿越 | `_send_static` 通过 `commonpath` 限制只能落在 `static/` 目录 |
| 金额精度漏洞 | `Decimal` + 8 位量化 + 拒绝 NaN/Infinity |
| 链篡改 | 区块哈希 + previous_hash 链接 + `validate_chain` 检测 |

## 9. 性能与扩展性

### 9.1 性能目标

- 单条限价单提交时间：亚毫秒级（仅依赖 GIL + SQLite 写锁）。
- 撮合复杂度：每笔订单 O(N)，N 为对手簿剩余订单数。
- 链校验：O(B + T)，B 为区块数、T 为总成交数。

教学场景下对绝对性能无量化承诺。

### 9.2 扩展点

| 扩展方向 | 改造点 |
| --- | --- |
| 接入真实数据库（PostgreSQL） | 替换 `database.py` 与服务的 `_load_from_db` / `_persist_*` 方法 |
| 接入消息队列驱动撮合 | `MatchingEngine.place_*` 改为发送事件至队列，撮合服务异步消费 |
| 引入手续费 | 在 `_execute_trade` 中按比例扣减、转入手续费账户 |
| 引入更多订单类型（止盈止损） | 在 `OrderType` 与 `MatchingEngine` 撮合主循环中扩展条件分支 |
| 引入 token 过期 | `sessions` 表增加 `expires_at`，`get_user_by_token` 校验时间 |
| 多进程协调 | 替换为 PostgreSQL/Redis，加分布式锁；或将撮合改为单写多读架构 |

### 9.3 演化策略

- **保持模型层稳定**：`models.py` 与 `exceptions.py` 是契约边界，扩展功能尽量不修改既有签名。
- **持久化适配器化**：当前 `_load_from_db` / `_persist_*` 方法已隐式形成"仓储模式"，未来可拆分独立 Repository 类。
- **接入层薄化**：`gui.py` 处理 HTTP 与鉴权，业务规则全部下沉到服务层，便于替换为 Flask/FastAPI 等框架。

## 10. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 撮合中途异常导致部分状态写入 | 钱包余额与订单不一致 | 教学场景下接受；后续可引入数据库事务包裹 `_match_order` |
| SQLite WAL 文件残留 | 重置后状态混乱 | `clear_db` 仅清数据，不动 schema；必要时直接删除 `dex.db*` |
| 端口被占用 | 启动失败 | 8000~8019 自动顺延 |
| 鉴权 token 没有过期 | 长时间运行可能积累有效 token | 定期重置 / 后续扩展过期时间 |
| 浏览器缓存导致前端不更新 | 教学时学生看到旧 UI | 推荐强制刷新（Cmd/Ctrl+Shift+R） |
